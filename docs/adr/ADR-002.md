# ADR-002: Configuration System

## Context
Applications require external configuration (API keys, ports, toggles) to function correctly across different environments (development, testing, staging, production). Hardcoded values lead to security vulnerabilities and inflexible deployments (e.g., Docker containers that require rebuilding to change a timeout).

## Decision
We are adopting a centralized, strongly typed, environment-driven configuration system using **Pydantic Settings v2**. 
- Configuration values are separated into logical domain models (e.g., `APISettings`, `AIProviderSettings`).
- A singleton loader (`get_settings()`) cached via `@lru_cache` provides immutable access to the settings throughout the application.

## Rationale
- **Security**: Sensitive data is injected via environment variables and never checked into source control.
- **Type Safety**: Pydantic guarantees that configuration values are cast to their correct types (integers, booleans) at startup, preventing runtime crashes.
- **Immutability**: Configuration cannot be modified dynamically at runtime by application code, ensuring predictable behavior.

## Consequences
- All configuration changes must be routed through environment variables or `.env` files.
- Configuration loading is highly performant since it is cached on first access.
