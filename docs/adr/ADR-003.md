# ADR-003: Dependency Injection

## Context
When services explicitly instantiate their dependencies (e.g., `ai_client = FireworksClient()`), it creates tight coupling. This makes it impossible to substitute dependencies dynamically, which is critical for unit testing (where we need `MockAIClient`) and for future extensibility.

## Decision
We will employ **Dependency Injection (DI)** globally across the application.
- FastAPI's `Depends()` system will be utilized at the route level to inject use cases and configuration.
- Domain services will accept their dependencies via constructor injection (e.g., passing `AIProviderInterface` into the `CaptionGenerator`).
- No service will use global variables (except for stateless utility functions).

## Rationale
Dependency Injection inverts control. It allows the Application layer to compose objects dynamically, enabling massive flexibility. It aligns perfectly with the SOLID principles, specifically the Dependency Inversion Principle.

## Consequences
- Testing becomes significantly easier since components can be verified in total isolation.
- The instantiation graph (the code that wires everything together) becomes slightly more complex, likely handled by a dedicated `dependencies/` module.
