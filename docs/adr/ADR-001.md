# ADR-001: Clean Architecture

## Context
CaptionDB is intended to be a production-quality application that survives beyond its initial hackathon scope. It must be resilient to changing requirements, highly testable, and completely independent of any specific framework, database, or external API (such as Fireworks AI). Tightly coupling business rules to external dependencies or UI concerns leads to technical debt.

## Decision
We will adhere strictly to the **Clean Architecture** pattern. The backend is organized into four concentric layers:
1. **Presentation (API/Delivery)**: FastAPI routers, HTTP request validation.
2. **Application (Use Cases)**: Pipeline orchestrators and business process coordination.
3. **Domain (Core)**: Business entities, interfaces, and core AI processing logic.
4. **Infrastructure (Adapters)**: Implementations of external integrations (Storage, AI Providers, Databases, Configuration).

Dependencies must point inward. Outer layers can depend on inner layers, but inner layers must never depend on outer layers.

## Rationale
By decoupling business logic from infrastructure, we isolate the core value of the application. If we need to replace FastAPI with a CLI, or switch Fireworks AI for an alternative VLM, the Domain and Application layers require zero modification.

## Consequences
- Increased initial boilerplate and architectural overhead (e.g., creating abstract interfaces before implementations).
- Higher long-term maintainability.
- Trivial unit testing since infrastructure layers can be easily mocked.
